컴파일러가 코드를 최적화
문법 그 이상의 것

컴파일은 디버그를 한후 릴리즈를 하는데
디버그는 되는데 릴리즈 하면 프로그램이 죽는 경우가 있다

최적화 -> 의존성
의존성을 변수를 기준으로
연산할때 변수의 의존성을 잘 파악해야 한다 =
순차적 진행의 순서를 잘 생각해야 한다

구문 간의 의존성과 개연성이 없다면 동시성이 가능하다

의존성을 변수를 기준으로 판단한다

릴리즈 모드를 하는 순간 최적화가 진행된다
릴리즈에서는 불필요하다고 컴파일러가 판단 되면 과정이 생략 된다

최적화는 군살을 뺴는 것이다 = 논리적으로 매우 당연한 것을 제거한다

특정 변수(자료)에 대해 의존성이 존재하는 연산들을 구별

변수가 많은 것이 좋지 않은이유는
변수의 조합(경우의 수)가 많아 지기 때문에
변수는 적을 수록 좋다

최적화된 코드를 작성하기 위해 노력 <
컴파일러가 최적화하기 좋게 작성

최적화 방해 요소
1. 변수가 많은 경우 -> 최대한 변수의 개수를 줄인다
2. 포인터 사용 주의(자제), c++ 에서는 포인터를 참조로 바꾸게 된다
참조 = 상수화 된 포인터

const를 잘 써야 좋다

심볼릭 상수
변경 가능성이 있으며 연산의 기준점이 되는 변수를
의미가 명확하게 알 수 있도록 만들어진 상수

volatile : 컴파일러가 변수를 생략 되지 않고 변역을 하도록 한다
특정 변수에 의존성이 있는 모든 연산을 최적화 하지 않는다

Link
함수의 선언과 정의를 분리하여 다른 파일로 저장할경우
main 파일에서 함수가 실행되면 중간에 extern table을 통해
함수 정의 를 찾아 실행후 다시 main으로 돌아와 파일을 마무리한다
다른 파일에 동일 함수명의 정의가 있을경우 링크 오류 발생한다

extern 외부변수
외부 파일에 이미 '정의'되어있는 변수에 대해 '선언'만 한다

형식 재선언 : 자료형의 이름을 바꾸어 재 선언 하는일
사용하는 곳이 한정되어 있다
주의해서 사용해야 한다
형재선언은 생산성, 가독성을 떨어 뜨린다

정확한 기획 없이 사용하면 안된다

코딩 컨벤션(규칙) - 변수 이름 표기법

열거형 상수는 typedef 와 많이 사용된다
